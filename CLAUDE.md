# Asteroid Blaster — Claude Code Guide

## Project

Browser-based Asteroids clone. Pure ES modules, no framework. TypeScript + Vite + Vitest.
Deploy target: `/asteroids/` (relative base path set in `vite build --base=/asteroids/`).

```
npm run dev      # dev server
npm run build    # production build
npm run test     # vitest run --coverage
```

---

## Folder Structure

| Folder | Responsibility |
|---|---|
| `src/entities/` | Game object classes (Ship, Rock, Bullet, Gun, Viewport). Pure — no state or DOM dependencies. |
| `src/state/` | Central game state object + discriminated union dispatcher (`changeGameState`). |
| `src/update/` | Per-frame physics, collision detection, level completion. Calls `changeGameState` for mutations. |
| `src/render/` | Per-frame DOM diff-render: positions entities, plays sounds, updates FPS/score. |
| `src/ui/` | Screen components: start screen, end screen, level announcement, buttons. |
| `src/events/` | State machine lifecycle hooks (`onEnter`/`onExit`) for each `GamePhase`. |
| `src/input/` | Keyboard listeners and action aggregation for the ship. |
| `src/config/` | Constants only: entity specs, level progression, key bindings, frame rate, debug flags. |
| `src/utils/` | Pure functions: maths (collision, angles), physics (motion, velocity), random generators. |
| `src/assets/` | SVG definitions for ship and asteroids. |
| `src/types.ts` | Shared TypeScript types (`GamePhase`, `GameEntity`, `Position`, `Velocity`, etc.). |

`render/` handles game-entity rendering (rocks, ship, bullets per frame).
`ui/` handles screens and controls (created once on state transitions, not per frame).

## Game Loop

```
index.ts → requestAnimationFrame(step)
  └─ handleStateTransition()   // run onEnter/onExit if state changed
  └─ gameLoop(dt)
       ├─ gameLoopUpdate()     // physics, input, collisions, level checks
       └─ gameLoopRender()     // diff-based DOM updates
```

`dt` is a normalised delta-time multiplier (elapsed / TARGET_FRAME_MS, capped at 3).
The game loop only runs when `state === "playing"` or `"gameover"` (rocks animate during game over).

## State Management

All mutations go through `changeGameState(action)` in `src/state/game-state.ts`.

The dispatcher uses a **discriminated union** — do not destructure to `({ action, payload })` as it breaks TypeScript's type narrowing. Use `change.action` / `change.payload` directly inside the switch.

```typescript
changeGameState({ action: "state", payload: "playing" });
changeGameState({ action: "add rock", payload: rock });
changeGameState({ action: "score", payload: 100 });
```

`changeGameState` is a **pure state dispatcher** — no side effects (no DOM, no timers, no audio). Side effects belong in callers (`events.ts`, `gameloop-update.ts`).

All mutations are logged to `stateHistory[]` (capped at 10,000 entries) for debugging.

---

## State Machine

`GamePhase`: `"start"` | `"playing"` | `"paused"` | `"gameover"`

Valid transitions (anything else is a no-op in `handleStateTransition`):

```
start → playing
playing → paused
paused → playing
playing → gameover
gameover → start
```

Each transition runs `onExit(previous)` then `onEnter(next)`. Timer is reset on transitions into `"playing"`.

## Conventions

- **British English** in all identifiers, filenames, and comments (`maths` not `math`, etc.).
- **Co-located tests**: `foo.test.ts` lives alongside `foo.ts` in the same folder.
- **Test environment**: Vitest + happy-dom. Run with `npm test`.
- `dt` = delta time multiplier (used throughout `update/` and `entities/`).
- Entity IDs are strings generated by static class counters (`Rock.rockIDCounter`, `Bullet.bulletIDCounter`). These are used as DOM element IDs and object keys in `gameState.rocks` / `gameState.bullets`.
- `GameEntity` interface (`types.ts`) must be implemented by any entity that participates in the game loop: requires `id`, `update()`, and `boundary()`.
